# Anti-Olden: 职场沟通辅助 Agent

> 项目启动文档 v0.1 | 2026-04-16

---

## 1. 项目概述

### 1.1 问题背景

在职场中存在一类高沟通成本的协作对象（项目内部代号"老登"），其典型特征包括：高自我中心（high ego）、不倾听他人观点、习惯性否定但不提供建设性反馈、沟通风格消耗对方情绪但不推进实际工作。

与这类人的每次交互都是一次"情绪税"——你需要在管理自己的情绪反应的同时，组织有策略性的语言来推进工作。这种双重负荷导致大量心力浪费。

**数据支撑：** 这并非个例痛点。调研显示 51% 的 Gen Z 职场人使用 AI 帮助处理职场冲突和困难对话，56% 用 AI 获取与同事/经理沟通的建议。47% 的职场人已在使用 AI 起草书面内容，52% 用于编辑润色。然而现有工具（ChatGPT、Copilot 等）每次对话都从零开始——用户必须反复解释自己的角色、关系和沟通风格，这种"上下文重建税"（context re-establishment tax）是目前最被诟病的问题。详见 [调研文档](./current-ai-usage-for-workplace-communication.md)。

### 1.2 核心理念

**将情绪与策略分离。** Agent 不会被激怒、不会有情绪波动，它只关注一件事：如何用最合适的措辞把事情往前推一步。它是你的"沟通参谋"——你按下暂停键，和它商量怎么回，然后带着经过思考的回复回到战场。

**定位是"参谋"而非"代笔"。** 调研表明，重度 AI 辅助的消息只被 40-52% 的员工认为是真诚的，而低 AI 介入的消息这一比例为 83%。产品定位应强调"帮你沟通得更好"，而不是"替你沟通"——用户把 AI 当教练/编辑的效果远好于当代笔人。

### 1.3 产品形态：双轨并行

| 维度 | 桌面端 App | Claude Code / Desktop Skill |
|------|-----------|----------------------------|
| 定位 | 完整体验，独立产品 | 轻量集成，即插即用 |
| 交互 | 快捷键呼出，窗口式会话 | 在 Claude 会话中调用 Skill |
| Memory | 本地存储 + 向量检索 | Skill 目录下的文件系统 |
| 用户群 | 日常高频使用者 | Claude 生态用户、开发者 |
| 优先级 | 中期主力 | 短期快速验证 |

两条线共享核心逻辑：消息解析、人物画像、回复策略生成。**数据接入层也统一为 Lark CLI**——桌面端用 Rust `Command` 调用，Skill 用 Claude Code 的 Bash 工具调用。区别仅在 UI 层和 Memory 存储层。

## 2. 用户场景与流程

### 2.1 核心用户流程（桌面端）

```
飞书收到 @你 的消息 → 感觉"这条不好回" → 快捷键呼出 Anti-Olden
       ↓
Agent 自动拉取消息上下文（通过 Lark CLI）
       ↓
加载该对话人的 Memory（沟通风格、历史策略、雷区）
       ↓
Agent 展示：① 消息分析 ② 对方意图解读 ③ 推荐回复（含策略标签）
       ↓
用户可以：调整策略 / 修改措辞 / 追加要求（"加上上次会议结论作为依据"）
       ↓
满意后复制回复 → 粘贴到飞书发送
       ↓
Agent 记录本次交互，更新该人物的 Memory
```

### 2.2 核心用户流程（Skill 模式）

Skill 模式同样基于 Lark CLI 做数据接入，但利用 Claude Code 的 Bash 工具直接执行命令，不需要打包 MCP。用户可以用自然语言探索飞书空间，Skill 在需要时自动调用 CLI。

```
用户在 Claude Code 中触发 Skill
       ↓
用自然语言描述场景：
 - "看看「产品技术周会」群里张三最近@我的消息"
 - "昨天跟李四那个对话我想回一下，帮我想想怎么说"
 - "把我今天所有被@的消息都捋一遍，哪些需要重点回应"
       ↓
Skill 通过 Bash 调用 Lark CLI，拉取相关消息和上下文
       ↓
Skill 读取 memory/persons/{name}.md 加载人物档案
       ↓
输出分析 + 回复建议（支持多轮讨论措辞）
       ↓
用户确认后：
 - Skill 更新 memory 文件
 - 可选：通过 Lark CLI 直接发送回复（需用户确认）
```

**前置条件：** 用户已在本地安装 Lark CLI 并通过 `lark login` 完成 OAuth 授权。Skill 在首次运行时会检测并引导用户完成这一步。

### 2.3 Onboarding 流程

桌面端首次启动时引导用户完成：

1. **Lark CLI 认证** — 通过 OAuth 登录飞书账号，授权消息读取权限
2. **个人背景设置** — 你的角色、团队、工作职责（帮助 Agent 判断什么对你有利）
3. **默认回复风格** — 从预设中选择，后续可随时调整
4. **快捷键设置** — 设置全局唤起快捷键

## 3. 功能设计

### 3.1 消息解析引擎

**输入：** 飞书消息及其上下文（前后 N 条消息、所在群组信息）

**输出：**

- **消息摘要** — 对方在说什么、要求什么
- **意图分析** — 对方的真实目的（是在推卸责任？是在试探你的底线？还是确实在寻求信息？）
- **情绪判断** — 对方的情绪状态和攻击性程度
- **上下文关联** — 这条消息跟之前的哪些讨论有关

### 3.2 回复策略系统

提供三种预设策略，用户可在设置页默认选择一种，也可每次对话时切换：

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| **柔和推进** | 先认可对方观点，再温和引导到正确方向 | 对方需要被尊重感、ego 很高 |
| **就事论事** | 聚焦事实和数据，不回应情绪，只推进问题 | 对方习惯性否定但无实质论据 |
| **强势推回** | 直接用流程、规则、历史记录反驳 | 对方在推卸责任或越界 |

随着 Memory 积累，Agent 会自动推荐策略："上次跟这个人用就事论事效果不错，建议继续。"

### 3.3 人物 Memory 系统

Memory 是这个产品的长期护城河。每个联系人维护一份持续更新的档案：

**Memory 维度：**

- **沟通风格画像** — 此人偏好直接还是迂回？喜欢数据还是叙事？决策快还是慢？
- **雷区与敏感点** — 哪些话题或措辞会触发对方的防御反应
- **有效策略记录** — 历史上哪些回复方式推进了事情，哪些适得其反
- **组织关系** — 此人在组织中的角色、影响力、与你的汇报关系
- **行为模式** — 比如"周一上午情绪差"、"被抄送领导后会配合"等观察

**Memory 更新机制：**

- **自动提取** — 每次交互后，Agent 从对话内容和用户的策略选择中提取信息
- **用户标注** — 用户可以主动补充："这个人其实是 VP 的人"、"上次这样回他配合了"
- **衰减机制** — 长期未验证的观察逐渐降低权重，避免过时信息误导

### 3.4 会话管理

**工作会话：** 每次处理一条消息就是一个工作会话。会话内用户可以和 Agent 多轮讨论措辞。

**会话历史：** 所有工作会话按人物和时间归档，方便回顾"上次跟张三那个事我是怎么回的"。

**会话与 Memory 的关系：** 会话是 Memory 的原始素材。Agent 从会话中提取模式，写入 Memory；Memory 反过来指导下一次会话的策略。

### 4.2 技术选型

#### 桌面端框架：Tauri v2

选择理由：

- 安装包体积小（约 600KB 起步 vs Electron 的 150MB+），适合工具型产品
- Rust 后端性能优秀，适合本地 Memory 检索和消息处理
- 支持全局快捷键、系统托盘、多窗口等桌面端能力
- v2 的 IPC 基于 custom protocol，性能优于 v1
- 安全模型基于 ACL，可以精确控制前端对系统资源的访问

前端框架建议在 React / Vue / Svelte 中选择。考虑到工具型产品 UI 不复杂，Svelte 的轻量和简洁可能是好选择。

#### 飞书集成：Lark CLI

[Lark CLI](https://github.com/larksuite/cli) 是飞书官方维护的命令行工具，关键能力：

- **消息读取** — 支持查看聊天历史、线程、搜索消息
- **消息发送** — 支持发送和回复消息（未来可考虑直接从 Agent 发送）
- **认证** — 支持 OAuth 交互式登录、设备码流程、身份切换（用户/机器人）
- **200+ 命令** — 覆盖 Messenger、Docs、Calendar 等核心业务域
- **结构化输出** — 输出支持 JSON 格式，方便程序解析

集成方式：Tauri 的 Rust 后端通过 `Command` 调用 Lark CLI 进程，解析 JSON 输出。

**注意事项：** Lark CLI 主要设计为"发送端"，实时消息监听能力有限。如需实时推送，可能需要结合飞书 Open API 的 WebSocket 事件订阅（`lark-event` skill）。但对于 v1（手动触发模式），CLI 的消息读取能力已经足够。

#### LLM 选型对比

| 维度 | Claude API | OpenAI API | 本地模型 (Ollama等) |
|------|-----------|------------|-------------------|
| **中文能力** | 优秀 | 优秀 | 取决于模型，中等偏上 |
| **长上下文** | 200K tokens | 128K tokens | 通常 8-32K |
| **策略理解** | 擅长细腻的语境和策略分析 | 擅长结构化输出 | 能力有限 |
| **隐私** | 数据发送到云端 | 数据发送到云端 | 完全本地，最佳隐私 |
| **成本** | 按 token 计费 | 按 token 计费 | 一次性硬件成本 |
| **延迟** | 1-3 秒 | 1-3 秒 | 取决于硬件，可能更快 |
| **Skill 生态** | 与 Claude Code/Desktop Skill 天然兼容 | 需要额外适配 | 需要额外适配 |

**建议：** 采用 provider-agnostic 的抽象层设计。定义统一的 `LLMService` 接口，后端实现 Claude / OpenAI / Local 三种 adapter。v1 先用 Claude API 跑通（与 Skill 路线一致），后续按需扩展。

#### Memory 存储：本地优先方案

基于调研，以下是三个值得考虑的方案：

| 方案 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| **Mem0 (开源版)** | 最成熟的 Agent Memory 框架，支持三层 memory（user / session / agent），后端可选 ChromaDB | 社区活跃（48K+ stars），API 设计成熟，支持自动 memory 提取 | Python 生态，与 Rust 后端集成需要跨语言调用 |
| **SQLite + ChromaDB** | SQLite 存结构化数据（人物档案、策略记录），ChromaDB 做向量检索（语义搜索历史对话） | 完全本地，无依赖，可控性强 | 需要自己实现 memory 提取和管理逻辑 |
| **MemPalace** | 基于 SQLite + ChromaDB 的本地优先 memory 系统，自带 19 个 MCP tools | 开箱即用，专为 Claude Code 生态设计 | 较新，社区较小，定制空间待验证 |

**建议：** 桌面端用 **SQLite + ChromaDB** 自建，控制力最强，且都有 Rust binding。Skill 侧用文件系统（markdown 文件），保持简单，与 Claude 的 memory 机制一致。

## 5. Skill 设计（Claude Code / Desktop）

### 5.1 设计原则

**Lark CLI 作为唯一数据接入层。** Skill 不包装 MCP、不自建 SDK，直接利用 Claude Code 的 Bash 工具调用已安装的 Lark CLI。好处：

- Lark CLI 是飞书官方工具，200+ 命令 + 20+ 内建 AI Agent Skills，能力完整且持续维护
- 认证由 `lark login` 一次性处理，Skill 不接触凭证
- Skill 只专注"沟通分析 + 人物 Memory"这一层增值逻辑，职责清晰
- 桌面端和 Skill 复用同一套 CLI 调用逻辑，只是 Skill 用 Bash 调，桌面端用 Rust `Command` 调

**一个 Skill，不拆分。** 虽然"飞书通用访问"理论上可以独立成一个 Skill，但 Lark CLI 本身就是为 Agent 设计的，Claude Code 通过 Bash 已经可以直接理解和使用。anti-olden Skill 做的事是在其上加一层"职场沟通策略 + 人物记忆"的专业能力，不需要再抽一层出来。

### 5.2 Skill 目录结构

Skill 源代码在仓库内放在 `skills/anti-olden/` 下，与桌面端源码（未来加入）并列。安装时整个 `anti-olden/` 目录会被复制/软链到 `~/.claude/skills/anti-olden/`。

```
<repo root>/
└── skills/
    └── anti-olden/
        ├── SKILL.md                    # Skill 入口：定义触发条件、核心能力
        ├── references/
        │   ├── lark-cli-cookbook.md    # 常用 Lark CLI 命令配方（读消息、搜群等）
        │   └── olden-patterns.md       # "老登"行为模式速查（可选加载）
        ├── prompts/
        │   ├── analyze.md              # 消息分析 prompt
        │   ├── reply-soft.md           # 柔和推进策略 prompt
        │   ├── reply-factual.md        # 就事论事策略 prompt
        │   └── reply-push.md           # 强势推回策略 prompt
        ├── memory/
        │   ├── index.md                # Memory 索引（哪些人有档案）
        │   └── persons/                # 每个人一个 markdown 文件
        │       ├── zhang-san.md
        │       └── li-si.md
        └── templates/
            └── person-profile.md       # 人物档案模板
```

### 5.3 SKILL.md 触发方式

Skill 通过以下关键词/场景触发：

- 用户提到飞书消息、群聊、@我、回复、沟通策略
- 用户说"怎么回"、"帮我想想怎么说"、"这条消息不好回"
- 用户主动 `/anti-olden` 调用

触发后，Skill 会：

1. **检查前置条件** — 通过 `lark --version` 确认 CLI 安装；通过 `lark whoami` 确认登录状态。如缺失，引导用户完成
2. **理解场景** — 根据用户自然语言，决定是拉消息、分析还是生成回复
3. **调用 Lark CLI** — 通过 Bash 执行对应命令，获取 JSON 输出
4. **匹配 Memory** — 根据对话人姓名或 ID 查 `memory/persons/`
5. **输出分析与建议**
6. **更新 Memory**

### 5.4 Skill 工作流示例

场景：用户说"我今天被张三 @ 了两次，帮我看看怎么回"

```
Step 1: Bash 执行 `lark messenger mentions --since today --format json`
        → 获取今天被 @ 的消息列表
Step 2: 解析 JSON，找到来自"张三"的消息，展示给用户确认是哪一条
Step 3: Bash 执行 `lark chat history --chat-id <id> --before <msg-id> --limit 20`
        → 获取该消息上下文
Step 4: Read memory/persons/zhang-san.md → 加载张三的档案
Step 5: 使用 prompts/analyze.md 分析消息
Step 6: 向用户展示分析结果 + 推荐策略 + 生成的回复
Step 7: 用户多轮调整措辞，确认后：
        - Edit memory/persons/zhang-san.md 更新档案
        - （可选）提示用户：是否需要 `lark messenger reply` 直接发送？
          注意：发送属于写操作，必须显式经用户确认
```

### 5.5 Memory 文件格式（Skill 版）

```markdown
---
name: 张三
lark_user_id: ou_xxxxxxxxxxxxxxx
department: 技术部
role: 高级工程师
last_updated: 2026-04-16
interaction_count: 12
---

## 沟通风格
- 偏好直接表达，不喜欢铺垫
- 习惯用反问句否定他人观点
- 对数据和事实比较认可，纯逻辑说服效果好

## 雷区
- 不要质疑他的技术判断，会触发强烈防御
- 避免在群里公开反驳，私聊效果更好

## 有效策略
- 2026-04-10: 用了"就事论事"策略，引用了测试数据，他接受了方案 ✓
- 2026-04-05: 用了"柔和推进"，先认可他的担忧再给替代方案，效果一般 △

## 关系备注
- VP 直接下属，在团队有较大话语权
- 和产品经理李四有历史矛盾，涉及他们的事要谨慎
```

注：`lark_user_id` 是关键字段，用于跨消息追踪同一个人（姓名可能有重名）。

### 5.6 关键设计决策

**写操作的安全边界** — Skill 可以读消息、读上下文，但所有写操作（发送消息、修改群设置等）必须经用户显式确认。默认行为是"生成回复文本让用户自己复制粘贴"，不自动发送。

**CLI 命令的可发现性** — `references/lark-cli-cookbook.md` 里预写好常用命令模板（拉群列表、搜消息、拉历史等），避免每次都让 Claude 从头摸索命令语法。Skill 运行时优先参考 cookbook，cookbook 里没有的再查 `lark --help`。

**Memory 命名冲突** — 姓名相同的人用 `zhang-san_ou_abc123.md` 这样的文件名区分，或在 `memory/index.md` 维护姓名到文件的映射。

## 6. 开发路线

### Phase 0: Skill 原型（1-2 周）

**目标：** 最快速度验证核心价值——Agent 能否通过 Lark CLI 拉到上下文，并给出有用的回复建议

- 本地安装 Lark CLI 并完成 `lark login`
- 编写 SKILL.md：定义触发条件、前置检查、工作流
- 编写 `references/lark-cli-cookbook.md`：常用命令配方（拉@消息、拉群历史、搜消息）
- 编写三种策略的 prompt 模板
- 设计人物档案模板和 `memory/` 目录结构
- 在 Claude Code 中用自己真实的飞书场景测试
- 迭代 prompt 和 cookbook

**交付物：** 可用的 Skill 目录，可直接安装到 Claude Code

**关键验证点：**
1. Claude 能否正确理解自然语言并翻译成 Lark CLI 命令
2. CLI 返回的 JSON 是否足够让 Claude 理解消息上下文
3. 生成的回复建议是否真的"比我自己想得好"

### Phase 1: 桌面端 MVP（4-6 周）

**目标：** 完成桌面端基本可用

- Tauri v2 项目搭建，实现全局快捷键呼出
- 接入 Lark CLI，实现消息上下文拉取
- 接入 LLM API（先选一个 provider）
- 实现基础 Memory 存储（SQLite 先行，暂不引入向量检索）
- 基础 UI：消息展示、策略选择、回复编辑、一键复制

### Phase 2: Memory 增强（2-3 周）

**目标：** Memory 系统达到可用水平

- 引入 ChromaDB 做语义检索
- 实现自动 Memory 提取（从会话中自动归纳人物特征）
- Memory 衰减机制
- 人物档案可视化界面

### Phase 3: 体验打磨（持续）

**目标：** 从"能用"到"好用"

- 策略推荐引擎（基于 Memory 自动推荐策略）
- 会话历史搜索和回顾
- 多 LLM provider 支持
- Onboarding 流程优化
- 飞书实时消息推送（WebSocket 事件订阅）

## 7. 市场验证与竞品分析

> 本节数据来源：[调研文档](./current-ai-usage-for-workplace-communication.md)

### 7.1 痛点验证

调研确认了三个核心痛点与 Anti-Olden 的产品假设高度吻合：

**痛点 1: 上下文重建税（Context Re-establishment Tax）。** 这是现有 AI 沟通工具最大的用户抱怨。每次新对话都要从头解释角色、关系、沟通风格。用户自发构建了各种 workaround（风格指南文档、上下文摘要、Custom GPTs），这些手动方案本身就是需求信号——Anti-Olden 的人物 Memory 系统正是对这一需求的产品化回应。

**痛点 2: 不理解人际动态。** 没有任何现有工具能理解"给你的 skip-level 经理写消息和给直接下属写消息需要不同策略"。Anti-Olden 的人物档案 + 策略推荐机制针对的就是这个空白。

**痛点 3: "企业八股文"问题。** AI 默认生成过度正式、模板化的措辞，用户明确称 ChatGPT 为"corporate bullshit machine"。收件人越来越能识别 AI 生成的语言，信任度下降。这强化了 Anti-Olden "参谋而非代笔"的定位——目标是帮用户想清楚怎么说，而不是替用户说。

### 7.2 用户画像补充

调研揭示了两类 Anti-Olden 的潜在高价值用户：

**高情绪成本场景用户。** 一个常见的工作流被称为"情绪宣泄+改写"：用户先写一封完全不加掩饰的愤怒草稿，把情绪全部释放出来，然后让 AI 改写成专业版本。这恰好是 Anti-Olden 的核心场景——差别在于 Anti-Olden 还加了人物记忆和策略选择。

**Gen Z / 早期职场人。** 76% 的 Gen Z 职场人使用 AI chatbot，75% 用 AI 分析消息语气——他们不仅用 AI 写消息，还用 AI 理解收到的消息意味着什么。47% 更依赖 AI 建议而非经理指导。这说明他们缺的不只是"写作能力"，而是"读懂职场社交语境的能力"。Anti-Olden 的消息分析功能（意图解读、情绪判断）对这个群体尤其有价值。

### 7.3 竞品格局

| 竞品 | 定位 | 与 Anti-Olden 的差异 |
|------|------|---------------------|
| **ChatGPT / Claude（通用）** | 通用 AI 助手，paste-and-prompt | 无持久记忆、无人际动态理解、每次从零开始 |
| **WriteMail.ai** | Gmail Chrome 插件，320K+ 用户 | 仅限邮件，无对话上下文，无人物画像 |
| **MailMaestro** | Outlook/Gmail 插件，语气预设 | 局限于邮件，无策略分析 |
| **Superhuman** | 键盘驱动的邮件客户端 | AI 仅做摘要和草稿，无人际策略层 |
| **Microsoft Copilot** | M365 集成，线程摘要 + 草稿回复 | 最大威胁但：$99/user/月企业定价，上下文限于 M365，不建模人际动态 |

**关键发现：没有任何现有产品解决"持久的人际上下文"问题。** 它们自动化的是邮件机制（起草、摘要、排程），而不是维护你的职场关系模型——谁跟你有什么沟通历史、对特定人该用什么策略、什么措辞会触发什么反应。这正是 Anti-Olden 的差异化空间。

### 7.4 产品定位启示

调研中一个关键发现值得写入产品原则：**重度 AI 辅助的消息被认为真诚的比例仅 40-52%，而低 AI 介入的消息为 83%。** 这意味着：

- Anti-Olden 不应定位为"自动回复机器人"，而是"帮你想清楚怎么回的参谋"
- 默认行为是生成建议供用户参考和修改，而不是直接发送
- UI 上应鼓励用户在 Agent 建议基础上用自己的话重写，而不是原封不动复制
- 这也是为什么"一键复制"比"一键发送"更合适作为默认操作

## 8. 开放问题

以下问题暂不阻塞开发，但需要在过程中逐步回答：

1. **隐私与合规** — 飞书消息内容发送到云端 LLM 是否有合规风险？是否需要本地模型作为备选？
2. **Memory 冷启动** — 新联系人没有任何 Memory 时，Agent 的建议质量如何保证？是否需要用户手动填写初始档案？
3. **多人群聊场景** — 群里有多个"老登"同时 @ 你时，如何处理多人上下文和策略冲突？
4. **效果度量** — 如何衡量 Agent 的回复建议是否真的有用？是否需要用户对每次建议打分？
5. **Skill 与桌面端的 Memory 同步** — 如果同时使用两条线，Memory 如何保持一致？
6. **分发与可见性** — Skill 走公开 Git 仓库分发（`prompts/` 等核心 IP 必然公开）、私有仓库 + 安装脚本、还是拆两个仓库（桌面端私有 + Skill 独立公开发行版）？
   - **当前决策（2026-04-16）：** 内测期间仓库保持私有，只自用。分发路径延后到 Phase 0 验证完成后再定。
7. **付费意愿与隐私敏感度** — 调研未找到用户为"持久职场上下文"付费的数据，也未评估用户分享人际关系细节给 AI 工具的舒适度。微软 Copilot 2026 年 2 月暴露机密邮件的安全事件说明这是真实风险。
8. **AI 依赖性风险** — 30% 的日常 AI 用户报告经常感到冒名顶替综合征，15% 表示因有 AI 而不再在会议上提问。Anti-Olden 应避免让用户完全依赖 Agent 而丧失自身沟通能力——可考虑在 Memory 中展示用户自己沟通能力的成长轨迹。
