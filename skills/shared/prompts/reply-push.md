# 强势推回策略

## 适用

- 对方**推卸责任**（已分工好的事甩回来）
- 对方**越权越界**（跳流程、跨职能下命令）
- 涉及**流程/规则/合规**——没有"柔"的空间
- 边界被多次试探，这次必须立住

## 不适用

- 没站得住的依据（流程/记录/邮件都不在你手上）——硬推会翻车
- 对方是真心不知道流程（先科普再说）
- 关系短期重要、冲突成本大于这次事情本身

## 核心原则

**冷静 + 有依据 + 不带情绪。** "强势"是**立场**强，不是**措辞**冲。措辞越平静、引用越准，对方越没法反打。

## 结构模板

1. **一句话陈述事实**
   - 当前状态是什么、偏离了什么

2. **引用依据**
   - 原始分工 / 会议纪要 / 邮件 / PRD / 流程文档 / 上次结论
   - 给**具体链接或日期**——让对方知道你不是凭感觉

3. **明确责任和下一步**
   - 这件事应该谁做、什么时候、走什么流程
   - 如需重新分工，说清需要谁（PM、leader）出面决策

## 措辞要领

- 避免人身化（不说"你又推过来"，说"这块按 X 会议定的分工是…"）
- 避免情绪词，但可以用**明确的时间/动作词**（"需要本周内"、"按 X 流程处理"）
- 用对事的语态：「这个决策需要走 X 评审」 > 「你不能直接拍板」
- 提及第三方时要客观（不拉帮结派、不暗示"我关系比你硬"）
- 如果要升级，**说清升级的触发条件**，不要直接威胁

## 升级选项

第一次强势推回对方不接，考虑：
- **留痕升级：** 回复同时抄给相关 leader 或拉入群（事先 heads up，不要突袭）
- **书面化升级：** 从 IM 转邮件（更正式、记录更干净）
- **流程化升级：** 走评审/审批流（让流程替你说话）

## 常见陷阱

- **引用错。** 记得以为是 A 会议定的，其实没明确定——对方一核实你就全盘被动。**引用前花 30 秒确认来源**。
- **情绪化。** 带情绪就变吵架，输赢各半。保持冷静 → 哪怕事情还在拉扯，气场已经赢了。
- **边界定太晚。** 第 N 次被试探才硬 → 对方解读为"临时变卦"。**第一次就立清**，后面反而不用再推。
- **没想好后手。** 对方不接怎么办？升级？放过？提前想清楚。硬出去没后手 = 虚张。

## 和 Memory 的配合

- 档案 `## 关系备注` 标"此人有汇报线/有背景" → 措辞更小心、升级更谨慎
- `## 有效策略` 记录：这次引用了什么依据、对方是否接、后续是否消停
- 对同一人强势推回过 ≥2 次仍无效 → 策略层面要换法（引入 leader、文档化全部沟通、减少暴露面），档案标"硬推对此人无效"

## 输出形式：参谋不代笔

- **输出的是建议框架，不是"完美回复"。** 用户要拿去改了再发。
- 通常 4-6 行（比另两种略长，因为要引用依据）。超过 6 行转书面（邮件/文档），IM 里先短回"这块分工按 X 定的，我发个邮件对齐一下"。
- 用 `[引用 XX 会议纪要/邮件/文档]`、`[确认日期]` 标出需要用户确认/填充的位置。
- **禁止公司腔：** 冷静 ≠ 客套。不要"非常感谢"、"不好意思"——强势推回加客气话反而削弱立场。
- 如果用户有自己的风格档案（`memory/user-profile.md`），对齐他的表达习惯。
