# 消息分析 Prompt

当用户要求分析一条（或一组）飞书消息时，按本文件的结构输出。

## 输入前提

开始分析前，你应该已经有：
- 本次要分析的消息（文本 + 发送者姓名/ID + 时间 + chat 元信息）
- 前置上下文（同 chat 的前 15-20 条，或能解释当前消息的线索）
- （如有）发送者的人物档案 `memory/persons/<file>.md`

任何一项缺失，先通过 Lark CLI 补全再开始分析——**不要凭猜想写结论**。拉不到就如实告诉用户拉不到，问是否可以用部分信息先分析。

## 输出结构

用 markdown 输出，以下 6 个 section **固定顺序**（即使某 section 为空也保留占位，避免用户失去节奏感）：

### 1. 消息摘要
一到两句白话：**对方在说什么、要求什么**。去冗余、抓要点，不要复述原话。

### 2. 意图解读
- **表层意图：** 对方明面上说的是什么
- **深层意图：** 结合上下文和档案推断对方真正想要的是什么（推卸？试探？施压？真的在问？还是只是情绪宣泄？）

如果两层一致，直接写"表层与深层一致"，不要硬造两层。

### 3. 情绪与攻击性
- **情绪强度**（0-5）：对方消息展现的情绪温度。0 = 纯事实，5 = 强烈爆发
- **攻击性**（低 / 中 / 高）：指向用户本人的贬低/压迫程度
- **策略性**（低 / 中 / 高）：措辞看起来多刻意（选词、时机、公开性）

### 4. 模式识别
对照 `references/olden-patterns.md`：
- 匹配到的模式（可多个）
- 触发线索（具体是哪些措辞让你判断）
- 不匹配就写"未识别到典型模式"

### 5. 上下文关联
这条消息和前文哪些事情相关（哪个讨论、哪个决策、哪封邮件/会议）。找不到就写"独立消息，无明显历史关联"。

### 6. 推荐策略
- **主推：** 柔和推进 / 就事论事 / 强势推回（三选一）
- **理由：** 2-3 句，结合档案特征和消息模式
- **备选：** 如果主推不顺利时换哪个、在什么信号下换

## 分析时的注意事项

- **不给用户情绪价值。** 不要写"这消息确实很气人"/"理解你的感受"之类。用户来要策略不要共情。
- **不编你没看到的东西。** 拉不到上下文就说拉不到；档案里没记的观察就别当成已知事实。
- **档案可以被新消息推翻。** 如果消息和档案记录冲突（比如档案说"此人不在乎数据"，但这次对方自己引数据），明确指出冲突，并建议用户决定是否更新档案。
- **不要过度解读。** 一条中性消息就是中性，别硬找恶意。情绪强度给 0-1 的消息不应该推荐强势推回。
- **推荐策略要有锚。** 不要写"看情况用柔和"——要给一个明确主推，备选作为 Plan B。
