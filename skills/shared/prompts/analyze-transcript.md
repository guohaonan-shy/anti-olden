# 会议转录分析 Prompt

当 comm-brief 从某场飞书会议的 transcript 为目标人物沉淀 memory 时，按本文件结构输出。

## 为什么要分析会议转录

**群里的人是他愿意让你看的形象，会议里的人才是他真实的尾巴。** 一些人在消息里克制得体，但在线上会议或线下沟通时才会暴露真实风格——语速、插话模式、面对反对意见怎么接、自信还是含糊、是否压人说话。这正是人物画像里消息数据漏掉的那一层。

## 输入前提

开始分析前，你应该已经有：
- 一份 transcript 文件（格式：说话人姓名 + 时间码 + 逐字内容，见下面"格式约定"）
- 目标人物的 `open_id`（从 `lark-cli contact +search-user` 拿到，用于验证发言人归属——transcript 里的说话人名是 display name，光凭名字可能有同名误伤）
- （如有）目标人物的现有档案 `../memory/persons/<file>.md`
- （如有）目标人物的 `user-profile.md`（你自己的画像，理解"你"在这场会议里的视角）

任一项缺失，先补全再分析。transcript 拉不全时如实告诉用户"只拿到 X 分钟"，不要凭想象补。

## Transcript 格式约定

`vc +notes --minute-tokens <token> --output-dir tmp/transcripts/<token>/` 下载的 `transcript.txt` 结构：

```
<ISO 时间>|<总时长>

关键词:
<AI 抽取的关键词列表>

<说话人英文名>(<说话人中文名>) HH:MM:SS.ms 
<发言内容（一行或多行，原始逐字，含填充词/打断/重复）>

<说话人英文名>(<说话人中文名>) HH:MM:SS.ms 
<下一段发言>
...
```

注意：说话人标签是**显示名**，不是 `open_id`。如果要严谨归属到 `persons/<X>.md`，先用 `lark-cli contact +search-user --query "<中文名>"` 验证，确认 open_id 跟目标一致再往档案里写。

## 输出结构

用 markdown 输出，按顺序给以下 6 个 section。即使某条目空也保留占位，让用户有节奏感。

### 1. 会议元信息

一到两句：时间、总时长、主题（从关键词 + 开头几段推断）、参会人列表（去重后）。

### 2. 目标对象的发言画像

这是核心——专门挖"**线下尾巴**"：

- **发言密度**：总时长里他说话大约占多少分钟 / 多少条 turn
- **语速与打断**：语速偏快/中/慢？有没有打断别人？有没有被打断后抢回？
- **自信度**：用"我觉得"/"我不确定"/"肯定是"/"必须"等措辞占比、hedge vs assert 的倾向
- **技术深度**：他抛细节还是抛结论？会不会被追问时立刻暴露空洞？
- **情绪线**：全程平稳 / 有明显情绪拉升点（标具体时间码 + 引文）
- **推进 vs 消耗**：他的发言是在把议题往前推，还是在绕/放大/转移焦点？

**每条判断都必须挂引文 + 时间码**，比如：
> `00:07:14` "这个我觉得不太对，我们上次不是说过了吗" —— 用 hedging ("我觉得") 开头但紧跟断言，典型半推进语气

不要写"他看起来很自信"这种无锚描述。

### 3. 线上 vs 线下对比

读已有 `persons/<X>.md` 的 `## 沟通风格` / `## 互动模式`。逐条判断：

- **强化**：transcript 观察**印证**档案已有描述 → 给时间码引文做佐证
- **冲突**：transcript 观察跟档案**不一致** → 明确说"档案记 X，会议里表现 Y"，让用户决定以哪个为准
- **新增**：会议暴露的某个特征档案完全没提 → 标"首次观察"

如果人物档案不存在，跳过对比，只在第 5 节建档。

### 4. 互动模式（target ↔ 其他参会人）

- 他跟**谁**说话最多？跟谁说话最少却被他频繁提及？
- 谁带节奏？谁在他说话时打断？他被打断后通常怎么反应？
- 有冲突或意见分歧时，他用什么语言模式处理（柔化 / 坚持 / 转移 / 撤回）？

每个模式也要挂时间码 + 引文。

### 5. 候选 memory 更新（diff-shape）

格式跟 comm-brief 主 pipeline 保持一致——**按目标文件 + section 分组**，增量语气：

```
## 候选写入：persons/<pinyin>_<ou 后6位>.md

### `## 沟通风格` (强化 / 冲突 / 新增)
- [会议:<会议标题>] <观察一句> —— 引文 `HH:MM:SS` "…"

### `## 互动模式` (新观察)
- [会议:<会议标题>] <观察一句> —— 引文 `HH:MM:SS` "…"

### `## 线下尾巴` (新增 section，如果档案现无此 section)
- [会议:<会议标题>] <会议里才暴露的特征，跟消息风格有反差的>
```

**关键约束：**
- 每条观察必须打 `[会议:<会议标题>]` 来源标签，跟消息来源 `[消息]` 区分
- 引文必须带时间码，短到能贴进一行但足以当证据
- 如果档案里已有 `## 沟通风格`/`## 互动模式`，**强化类**观察合并进去；**冲突类**要单独列出让用户判断；**新增类**优先塞进现有 section，塞不下再提议新开 section（如"线下尾巴"/"会议表现"）
- 宁缺毋滥：这场会议如果没挖到值得沉淀的东西（比如他就发言 2 句），直接说"本场无稳定观察"

### 6. 语气把关（自查清单）

写完 diff 后自查：

- [ ] 有没有把"他看起来 X"这种无证据描述写进观察？→ 删掉或补引文
- [ ] 会议里他被打断一次就写"他容易被压制"？→ 单次不是模式，标"单次观察，待验证"
- [ ] transcript 只有 3 分钟但挖了 5 条观察？→ 数据密度不够，砍到 1-2 条
- [ ] 引文有没有因剪短而歪曲原意？→ 看上下文，必要时多贴一条前后 turn
- [ ] 有没有给用户情绪价值（"他确实挺让人头疼"）？→ 删

## 分析时的注意事项

- **不给情绪价值，给观察。** 跟 `analyze.md` 一致，用户要画像不要共情。
- **逐字稿保留的 hedge 和打断是信号，不是噪声。** "嗯""那个""我就是""我觉得"这些填充词比"打磨过的书面语"更能暴露风格，别当废话跳过。
- **单场会议是样本，不是结论。** 一场开得顺的会不能断定他"协作能力强"；要挂"本场表现"或"单次观察"标签，等多场累积再收敛成稳定特征。
- **发言人名字有同名风险。** 写 `persons/<X>.md` 之前务必用 `contact +search-user` 把 transcript 里的中文名解析到 open_id，跟目标 open_id 一致再落档。
- **不编会议外的事。** 如果 transcript 里他提到"上次那事"但你不知道上次是什么，观察可以写"引用了外部上下文（'上次那事'，具体未知）"，不要自己补故事。
