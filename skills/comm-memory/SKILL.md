# 沟通记忆管理

## 你要做什么

管理三层 Memory 文件——用户画像、人物档案、群组上下文。AI 和用户是共同作者：用户手动标注的和 AI 提取的同等权威，混在同一文件里，不区分来源��

## 工具权限

- **Read** — 读 `../shared/memory/` 下所有文件
- **Write / Edit** — 写 `../shared/memory/` 下所有文件
- **AskUserQuestion** — 引导式建档问答、每次 A/B/C/D 确认都走此工具调用（不要用文本列"选 A/B/C/D"让用户打字）

不需要 Bash（不调 lark-cli）。不需要 WebSearch。

## Memory 结构

| 层 | 文件 | 说明 |
|---|---|---|
| 用户画像 | `../shared/memory/user-profile.md` | 我是谁 + 风格 + 禁忌 + 默认策略 |
| 人物档案 | `../shared/memory/persons/{person}.md` | 画像 + 雷区 + 互动模式 + 策略记录 |
| 群组上下文 | `../shared/memory/groups/{group}.md` | 氛围 + 关键人物在此群表现 |

设计文档：`../shared/memory/index.md`（**内容边界 / 运行时路由原则**，首次用时读一次）
只读模板：`../shared/memory/templates/`（schema 骨架，从不修改）
运行时写入路径：`user-profile.md` / `persons/<pinyin>.md` / `groups/<pinyin>.md`（这些文件 `.gitignore`，每人本地累积）

## 操作（全部对话式驱动）

### 所有写入前的通用模式

**不静默写 memory。** 每次 Write / Edit 前都遵循 `../shared/memory/index.md` 的"所有 memory 写入的通用规则"——调 `AskUserQuestion` 工具给 ABCD（含 C 修改循环 / D 不写）。

本 skill 特有的第 5 选项：**E. 重新问答** —— 用户对整段不满意，回到引导问答重新收集（只对"新建"流程适用，"编辑"不给这个选项）。

---

### 查看

用户说"看看张三的档案" / "张三什么情况" →
1. `ls persons/*.md` + 读 frontmatter 里的 `name` / `lark_user_id` 匹配；群档案同理 glob `groups/*.md`
2. Read 匹配到的文件
3. 给出结构化摘要，不要原样输出 markdown——用人话说

### 新建人物档案（引导式问答）

不要让用户自己填模板。通过问答收集，**问完后 AI 决定 section 结构**——不是机械映射到 template 的固定 section，而是根据用户的真实回答拼合适的结构：

- **参考骨架**：`templates/person-profile.md` 的 `## 沟通风格 / ## 雷区 / ## 互动模式 / ## 有效策略 / ## 关系备注`
- **但不是 schema**：如果用户描述的这个人有独特维度（"权力逻辑"、"爱好"、"家庭情况决定加班态度"……），应该**新开 section**，不要硬塞进骨架里
- 如果用户某个维度没料（说"不知道"/"没想过"/"没必要记"），**不写这个 section**——渐进累积，不填空

推荐的问答覆盖（次序灵活，不必每条都问）：

```
- "这个人叫什么？在哪个部门？什么角色？" → frontmatter
- "他平时说话是什么风格？直接还是绕？看数据还是看感觉？"
- "跟他说话有什么雷区吗？什么话不能说、什么场合不能反驳？"
- "你打算怎么跟他相处？（公事公办 / 笑里藏刀 / 真诚严谨 / 保持距离 / …）"
- "有什么关系背景？比如汇报线、跟谁有矛盾、在团队的影响力…"
- "还有什么关于他的关键信息你想记下的？"  ← 这条最重要，捕捉骨架之外的维度
```

**关键原则**：AI 收集完回答后，按内容本身决定 section——如果回答聚在骨架范围内就复用 template 的 section；如果有骨架覆盖不到的就新开 section。不要机械对应 Q→section。

**问答收集完成后不直接写盘**——进入"通用写入模式"（见顶部"所有写入前的通用模式"），展示完整 diff + 4 选项循环，用户选 A 才 Write。

写入路径：`../shared/memory/persons/<pinyin>.md`（重名时后缀 `_<ou_id 后 6 位>`）。**不修改 template**，每次新建都 Write 新文件。不维护集中索引——`lark_user_id` 存 frontmatter，后续查用 glob + frontmatter 匹配。

### 新建群组档案（引导式问答）

跟人物档案同原则——**参考骨架，不是 schema**。推荐问答覆盖（次序灵活）：

```
- "这个群叫什么？（如果知道 chat_id 也告诉我）" → frontmatter
- "这个群是什么氛围？正式还是轻松？领导在不在？"
- "这个群里谁是关键人物？他们在这个群跟平时有什么不同？"（可关联已有 person file，如"张三在这个群收着"）
- "在这个群发消息有什么要注意的？"
- "这个群还有什么特色？"  ← 捕捉骨架外的维度，比如"里程碑跟踪"、"瓜源属性"、"跨部门立场对立"等
```

问完由 AI 按真实内容决定 section 结构——`## 群氛围` / `## 关键人物` / `## 注意事项` 是起点参考；如果这个群有独特维度就新开。

### 用户画像首次建档

如果 `user-profile.md` 不存在，同样走"参考骨架"原则。推荐问答覆盖：

```
- "先了解一下你——你在团队是什么角色？负责什么？"
- "你平时在飞书上说话是什么风格？简洁直接？温和铺垫？数据驱动？随意口语？"
- "有没有什么措辞你绝对不会用？比如'您'、感叹号、emoji、长段…"
- "遇到不好回的消息，你一般倾向怎么处理？（就事论事 / 柔和推进 / 强势推回）"
- "还有什么关于你自己想让 Agent 知道的？"  ← 捕捉骨架外的维度
```

问完 AI 拼合适的 section 结构——起点参考 `templates/user-profile.md` 的 `## 我的角色 / ## 我的沟通风格 / ## 禁忌 / ## 默认策略偏好`；有独特维度就新开。

### 编辑已有档案

用户说"记一下张三这次配合了" / "我以后跟张三用就事论事" →

1. 定位文件：glob `persons/*.md`，按 frontmatter 的 `lark_user_id` 或 `name` 匹配（群档案同理 glob `groups/*.md`）
2. 定位到对应 section：
   - "这次配合了" → `## 有效策略`，追加：`2026-04-21: <场景> —— <策略> —— ✓ <复盘>`
   - "以后用就事论事" → `## 互动模式`，更新默认策略
   - "他其实不喜欢在群里被反驳" → `## 雷区`，追加观察
   - "他在产品群里很安静" → 对应 group file 的 `## 关键人物`
3. 进入"所有写入前的通用模式"（见顶部）——展示改动 diff + 4 选项循环。用户选 A 才 Edit。
4. 写入时同时更新 frontmatter `last_updated`（人物档案还有 `interaction_count`）

### 批量回顾

用户说"我有哪些人的档案" / "最近更新过哪些" →
1. `ls persons/*.md groups/*.md` + 读每个 frontmatter 的 `name` / `group_name` / `last_updated`
2. 按最近更新排序展示
3. 用户可以选某个深入看

## 设计原则

详见 `../shared/memory/index.md` 顶部。核心：
- **AI + 用户共同维护** — 不区分来源
- **人档案记稳定特征 + 我的姿态** — "张三 ego 高" + "我跟张三笑里藏刀"
- **群档案记场景特征** — "张三在这个群收着"
- **渐进累积** — 用户说"不知道"就跳过，不填空 section
- **可被推翻** — 新信息冲突就更新
- **编辑前确认** — 写入前给用户看将要写入的内容
